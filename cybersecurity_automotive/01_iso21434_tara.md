## 3. TARA — Step-by-Step

### Step 1: Item & Asset Definition
```
Item: Battery Management System (BMS)

Assets:
- Battery cell data (Confidentiality — leak = safety risk)
- Charge/discharge commands (Integrity — spoof = fire risk)
- BMS availability (Availability — DoS = vehicle inoperable)
```

### Step 2: Threat Identification (STRIDE)

| STRIDE | BMS Threat Example |
|--------|--------------------|
| **Spoofing** | Fake ECU sends discharge command over CAN |
| **Tampering** | Modify SOC algorithm via OBD-II |
| **Repudiation** | Deny unauthorized remote access |
| **Information disclosure** | Read cell voltages via XCP without authentication |
| **Denial of Service** | CAN bus flooding disables BMS communication |
| **Elevation of privilege** | Gain control of BMS via compromised telematics |

### Step 3: Attack Path Analysis

```
Attack: Remote disable of BMS

Attack path:
1. Attacker sends crafted SMS/MQTT to TCU via cellular
2. Exploits unpatched vulnerability in TCU HTTP server
3. Pivots to internal CAN via gateway (no CAN authentication)
4. Sends spoofed BMS discharge command at max rate
5. BMS executes command, battery deep-discharged → safety event

Prerequisites: Network access, TCU vuln, no CAN authentication
```

### Step 4: Feasibility Rating (ISO 21434 Annex B)

| Factor | Value | Score |
|--------|-------|-------|
| Attack window | Remote (always available) | High (0) |
| Expertise | Security researcher | Medium (3) |
| Knowledge | Partially public | Medium (3) |
| Equipment | PC + SIM card | High (0) |

**Overall Feasibility: High (0–3 → High)**

### Step 5: Impact Rating (SFOP)

| Dimension | Rating | Rationale |
|-----------|--------|-----------|
| Safety | Severe | Battery fire possible |
| Financial | Major | Recall + liability |
| Operational | Severe | Vehicle inoperable |
| Privacy | Negligible | No personal data |

**Worst-case Impact: Severe → Safety**

### Step 6: Risk Level

```
Risk = max(Impact) × Feasibility = Severe × High = CRITICAL
```

### Step 7: Cybersecurity Goals & Controls

| Goal | Control |
|------|---------|
| Authenticate CAN messages | SecOC with AES-128-CMAC |
| Protect TCU from remote exploitation | Regular security patches, FOTA |
| Prevent unauthorized XCP access | XCP seed-key authentication |
| Detect CAN flooding | On-board IDS, frame rate monitoring |

## 4. Security Architecture Patterns

### Defense in Depth
```
Internet ──→ [Firewall/Gateway] ──→ Telematics
                                        │
                              [Domain Gateway / Firewall]
                                        │
                    ┌───────────────────┼───────────────────┐
               [Powertrain          [ADAS              [Body
                CAN domain]          Ethernet]          LIN]
                    │
               [SecOC on all
                safety-critical
                CAN messages]
```

### Key Management
- Production keys provisioned in HSM during final vehicle assembly
- No default/debug keys in production firmware
- Key rotation via FOTA with signature verification

## 5. Security Testing Methods

| Method | Description | Tools |
|--------|-------------|-------|
| **Fuzz Testing** | Random/mutated inputs to find crashes | caringcaribou, boofuzz |
| **Protocol Analysis** | Capture and analyze bus traffic | Wireshark, CANalyzer |
| **Penetration Testing** | Emulate real attacker | Metasploit, custom scripts |
| **TARA Review** | Verify all threats have controls | Excel/Polarion |
| **Code Review** | Static analysis for crypto misuse | MISRA, Polyspace |
| **Fault Injection** | Voltage/clock glitching | ChipWhisperer |

## 6. Interview Q&A

**Q: What is the difference between functional safety (ISO 26262) and cybersecurity (ISO 21434)?**
> ISO 26262 addresses **random hardware failures and systematic errors** — unintentional faults. ISO 21434 addresses **intentional attacks by adversaries**. A system can be functionally safe but cybersecurity-vulnerable. Both are now required for modern vehicles, and they interact (e.g., a cyber attack can trigger a safety hazard).

**Q: Explain STRIDE in automotive context.**
> STRIDE is a threat model: Spoofing (fake sender identity), Tampering (data modification), Repudiation (deny actions), Information Disclosure (data leakage), Denial of Service (availability), Elevation of Privilege (unauthorized access level). In automotive: spoofing = fake CAN messages; DoS = bus flooding; elevation = gaining root on an ECU.

**Q: What is a residual risk?**
> After applying all cybersecurity controls, some risk remains — this is residual risk. ISO 21434 requires that residual risk be documented and accepted by management. If residual risk is too high, additional controls must be added or the feature must be redesigned.
