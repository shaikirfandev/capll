# AUTOSAR Basics Study Material
## AUTomotive Open System ARchitecture

---

## 1. What is AUTOSAR?

**AUTOSAR** (AUTomotive Open System ARchitecture) is an open, standardized automotive software architecture developed by a global partnership of OEMs, Tier-1 suppliers, and tool vendors.

**Goals:**
- Standardize ECU software structure and interfaces
- Enable software reuse and portability across ECUs/suppliers
- Separate hardware from application software
- Support complex, safety-critical systems at scale

**Two platforms:**
| Platform | Target | Language |
|---|---|---|
| **AUTOSAR Classic Platform (CP)** | Deeply embedded, real-time (engine, ABS, airbags) | C |
| **AUTOSAR Adaptive Platform (AP)** | High-compute, dynamic (ADAS, OTA, infotainment) | C++ (ara::*) |

## 2. AUTOSAR Classic Platform Architecture

```
┌──────────────────────────────────────────────────┐
│           Application Layer                       │
│  SWC    SWC    SWC    SWC    SWC    SWC           │
│ (Software Components)                             │
├──────────────────────────────────────────────────┤
│              RTE (Runtime Environment)            │
│    ← Virtual Function Bus (VFB) abstraction →    │
├───────────────────────┬──────────────────────────┤
│      Services Layer   │   ECU Abstraction Layer   │
│  (NvM, WdgM, ComM,   │  (IoHwAb, EcuAb)         │
│   DemM, Os, etc.)    │                            │
├───────────────────────┴──────────────────────────┤
│         Microcontroller Abstraction Layer (MCAL)  │
│  (ADC, PORT, DIO, SPI, CAN, LIN, Eth drivers)    │
├──────────────────────────────────────────────────┤
│              ECU Hardware                         │
└──────────────────────────────────────────────────┘
```

## 3. Key AUTOSAR CP Layers

### 3.1 Software Components (SWC)
- Units of application logic
- Communicate via **ports** and **interfaces**
- No direct hardware access (hardware-independent)
- Types: Application SWC, Sensor/Actuator SWC, Service SWC, ECU-Abstraction SWC

### 3.2 RTE (Runtime Environment)
- Automatically generated by AUTOSAR tools (EB tresos, Vector DAVINCI, ISOLAR)
- Implements **Virtual Function Bus (VFB)** locally on each ECU
- Routes inter-SWC and inter-ECU communication
- Generates type-safe API functions for SWCs

### 3.3 BSW (Basic Software)
Pre-configured, standard software modules:

| Layer | Modules | Purpose |
|---|---|---|
| **Services Layer** | OS, NvM, WdgM, ComM, Dem, Det, SchM | OS, storage, watchdog, comm management |
| **ECU Abstraction** | IoHwAb, EcuAb | Abstract ECU-specific hardware access |
| **MCAL** | ADC, PORT, DIO, SPI, CAN, Lin, Eth | Direct microcontroller peripheral drivers |

### 3.4 COM Stack (Communication Stack)
```
Application SWC
    |
   RTE
    |
   COM  ← PDU routing, signal packing/unpacking
    |
 PDU Router (PduR)  ← Route between COM stacks
    |
CanIf / LinIf / EthIf  ← Interface to physical bus
    |
CAN Driver / LIN Driver / Eth Driver (MCAL)
    |
Physical Bus (CAN / LIN / Ethernet)
```

## 4. SWC Ports and Interfaces

### Port Types
| Port Type | Description |
|---|---|
| **Sender-Receiver (S/R)** | Data flow – one SWC sends data, another receives |
| **Client-Server (C/S)** | Function call – client calls server operation |
| **Mode Switch** | SWC can switch or track mode groups |

### Interface Types
| Interface | Use |
|---|---|
| SenderReceiverInterface | S/R data elements |
| ClientServerInterface | C/S operations and arguments |
| ModeSwitchInterface | Mode declarations |
| ParameterInterface | Calibration parameters |

### Port Example (ARXML snippet)
```xml
<PORTS>
  <R-PORT-PROTOTYPE UUID="...">
    <SHORT-NAME>EngineSpeedIn</SHORT-NAME>
    <REQUIRED-INTERFACE-TREF DEST="SenderReceiverInterface">
      /Interfaces/If_EngineSpeed
    </REQUIRED-INTERFACE-TREF>
  </R-PORT-PROTOTYPE>
</PORTS>
```

## 5. Runnable Entities

A **Runnable** is the smallest unit of executable code within a SWC, similar to a function/task:

```c
/* AUTOSAR RTE-generated prototype */
void Rte_Runnable_SpeedControl_Run(void);

/* Implementation in SWC source */
FUNC(void, SpeedControl_CODE) SpeedControl_Run(void)
{
  float speed;
  float setpoint;
  
  /* Read data via RTE API */
  Rte_Read_EngineSpeedIn_EngineSpeed(&speed);
  Rte_Read_SetpointIn_TargetSpeed(&setpoint);
  
  /* Algorithm */
  float throttle = (setpoint - speed) * 0.1f;
  
  /* Write output via RTE API */
  Rte_Write_ThrottleOut_ThrottleCmd(&throttle);
}
```

### Runnable Triggers
| Trigger | Description |
|---|---|
| Timing event | Periodic (e.g., every 10 ms) |
| Data received | Triggered by incoming S/R data |
| Client invocation | Triggered by C/S call |
| Init event | At startup |
| Background | Low-priority background execution |

## 6. AUTOSAR OS (Operating System)

Based on **OSEK/VDX standard**:
- **Tasks**: Fixed-priority, preemptive scheduling
- **ISRs**: Category 1 (no OS API), Category 2 (OS API allowed)
- **Alarms**: Periodic task activation
- **Events**: Task synchronization
- **Schedule Tables**: Deterministic multi-rate task scheduling (common in automotive)

```c
/* Task definition example */
TASK(Task_10ms)
{
  SchM_Enter_SpeedControl();  /* Enter exclusive area */
  SpeedControl_Run();
  SchM_Exit_SpeedControl();
  TerminateTask();
}
```

## 7. AUTOSAR Diagnostic Services

| Module | Function |
|---|---|
| **DEM** (Diagnostic Event Manager) | Stores and manages DTCs, event memories |
| **DCM** (Diagnostic Communication Manager) | Handles UDS service requests (0x10, 0x22, 0x27...) |
| **DTC** | Diagnostic Trouble Code stored with snapshot data |

```
UDS Tool Request (0x19 ReadDTC)
       ↓
     DCM  ← processes UDS request, calls DEM
       ↓
     DEM  ← returns DTC list from event memory
       ↓
UDS Tool Response (DTC list)
```

## 8. AUTOSAR Adaptive Platform (AP)

Designed for **high-compute ECUs** (ADAS domain controllers, AI accelerators):

| Feature | Classic Platform | Adaptive Platform |
|---|---|---|
| Language | C | C++ (modern, C++14/17) |
| OS | AUTOSAR OS (OSEK) | POSIX (Linux-based) |
| Communication | COM/PDU | ara::com (SOME/IP) |
| Memory | Static allocation | Dynamic allocation allowed |
| Updates | Reflash required | OTA updates (UCM) |
| Use case | Engine, airbags, ABS | ADAS, gateway, infotainment |

### AP Key Functional Clusters
| Cluster | Purpose |
|---|---|
| **ara::com** | Service-oriented communication (SOME/IP) |
| **ara::diag** | UDS diagnostics |
| **ara::exec** | Execution management |
| **ara::log** | Logging |
| **ara::per** | Persistent Storage |
| **ara::ucm** | Update and Configuration Management (OTA) |
| **ara::phm** | Platform Health Management |

## 10. ARXML Files

ARXML (AUTOSAR XML) is the file format used for all AUTOSAR model data:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<AUTOSAR xmlns="http://autosar.org/schema/r4.0"
         xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance">
  <ADMIN-DATA>
    <LANGUAGE>FOR-ALL</LANGUAGE>
  </ADMIN-DATA>
  <AR-PACKAGES>
    <AR-PACKAGE>
      <SHORT-NAME>Interfaces</SHORT-NAME>
      <ELEMENTS>
        <SENDER-RECEIVER-INTERFACE>
          <SHORT-NAME>If_EngineSpeed</SHORT-NAME>
          <DATA-ELEMENTS>
            <VARIABLE-DATA-PROTOTYPE>
              <SHORT-NAME>EngineSpeed</SHORT-NAME>
              <TYPE-TREF DEST="ImplementationDataType">/DataTypes/float32</TYPE-TREF>
            </VARIABLE-DATA-PROTOTYPE>
          </DATA-ELEMENTS>
        </SENDER-RECEIVER-INTERFACE>
      </ELEMENTS>
    </AR-PACKAGE>
  </AR-PACKAGES>
</AUTOSAR>
```

## 11. Common Interview Questions

**Q1: What is the RTE in AUTOSAR and what does it do?**
> The Runtime Environment (RTE) is auto-generated middleware that implements the Virtual Function Bus locally on each ECU. It routes data and function calls between SWCs and handles inter-ECU communication transparently.

**Q2: What is the difference between Sender/Receiver and Client/Server interfaces?**
> S/R is for data flow — one SWC produces data that others consume. C/S is for function calls — a client invokes a specific operation on a server SWC and may get a return value.

**Q3: What is a Runnable entity?**
> The smallest schedulable unit in an AUTOSAR SWC. It's triggered by timing events, data reception, or client invocations and maps to OS tasks via the SchM (Schedule Manager).

**Q4: What is the difference between AUTOSAR Classic and Adaptive platforms?**
> Classic targets deeply-embedded, hard-real-time ECUs using C with static memory. Adaptive targets high-compute ECUs using C++ with dynamic features, POSIX OS, and SOME/IP for service-oriented communication.

**Q5: What does the DEM module do?**
> Diagnostic Event Manager — manages diagnostic events and DTCs. It receives fault reports from SWCs, applies debouncing and storage conditions, and stores DTCs with freeze frame/snapshot data for retrieval by DCM.
