## 1. Operating System Overview

The Sealion 7 uses **mixed-OS architecture** — different OS per compute domain based on safety requirements.

| Domain | OS | Rationale |
|--------|----|-|
| ADAS Domain Controller | **QNX Neutrino RTOS** (safety partition) + **Linux** (AI inference) | QNX: POSIX-compliant RTOS with ASIL D certification; Linux: rich ML frameworks |
| Powertrain ECU (VCU, BMS, MCU) | **AUTOSAR Classic CP** | Deterministic, hard-real-time, AUTOSAR standardized BSW stack |
| Chassis ECU (ESP, EPS, EPB) | **AUTOSAR Classic CP** | Safety-critical hard-real-time |
| Infotainment (Head Unit) | **Android Automotive OS 12+** (BYD DiLink skin) | Ecosystem, app support, OTA maturity |
| Instrument Cluster | **Linux** or **AUTOSAR Adaptive AP** | GPU-driven rendering, non-safety display |
| TCU | **Linux embedded** (Yocto-based) | Network stack, cloud connectivity |
| Gateway ECU | **AUTOSAR Classic CP** | Routing tables, security firewall |

### Sensor Fusion Algorithm Detail

```python
# Conceptual Extended Kalman Filter for camera-radar fusion

import numpy as np

class ObjectTracker:
    """
    Tracks a single object using camera (x,y,class) + radar (range,vel,azimuth).
    State vector: [x, y, vx, vy] in vehicle-centric coordinates
    """

    def __init__(self):
        self.x = np.zeros((4, 1))   # [x, y, vx, vy]
        self.P = np.eye(4) * 100    # Initial covariance (large uncertainty)
        self.F = np.eye(4)           # State transition (updated each step)
        self.Q = np.diag([0.1, 0.1, 1.0, 1.0])  # Process noise
        self.initialized = False

    def predict(self, dt):
        """Predict state using constant velocity model."""
        self.F[0, 2] = dt
        self.F[1, 3] = dt
        self.x = self.F @ self.x
        self.P = self.F @ self.P @ self.F.T + self.Q

    def update_radar(self, range_m, azimuth_rad, range_rate):
        """Update with radar measurement (polar → Cartesian Jacobian)."""
        r = range_m
        a = azimuth_rad
        rr = range_rate
        # Linearize measurement model H at current state
        H = np.array([
            [self.x[0,0]/r,  self.x[1,0]/r,  0, 0],
            [-self.x[1,0]/r**2, self.x[0,0]/r**2, 0, 0],
            [0, 0, np.cos(a), np.sin(a)]
        ])
        z = np.array([[r], [a], [rr]])
        R = np.diag([0.5, 0.01, 0.5])  # Radar noise
        self._kalman_update(z, H, R)

    def update_camera(self, x_m, y_m):
        """Update with camera-derived position (Cartesian)."""
        H = np.eye(2, 4)
        z = np.array([[x_m], [y_m]])
        R = np.diag([1.0, 1.0])  # Camera position noise (higher than radar for range)
        self._kalman_update(z, H, R)

    def _kalman_update(self, z, H, R):
        S = H @ self.P @ H.T + R
        K = self.P @ H.T @ np.linalg.inv(S)
        y = z - H @ self.x
        self.x = self.x + K @ y
        self.P = (np.eye(4) - K @ H) @ self.P
```

## 5. OTA (Over-the-Air Update Architecture)

```
BYD Cloud (OTA Server)
     │ HTTPS / MQTT (TLS 1.3)
     ↓
TCU (Telematics Control Unit)
     │ package download + signature verification
     ↓
Central Gateway (OTA Manager ECU)
     │
     ├── Campaign manager: schedule update window (SOC > 20%, vehicle parked)
     ├── Rollback manager: keep previous SW version in separate flash partition
     │
     ├── Head Unit OTA → Android A/B partition swap (seamless)
     ├── ADAS DCU OTA   → Dual-bank flash, update inactive bank, verify, swap
     └── ECU OTA (VCU/BMS/ESP/EPS) → UDS-based FOTA:
              ISO 14229 services:
              0x34 RequestDownload
              0x36 TransferData
              0x37 RequestTransferExit
              0x31 RoutineControl (verify checksum, activate SW)
```

**Safety during ECU OTA:**
- Vehicle must be in "Park" + HV off
- BMS must report SOC > 20%
- Watchdog timer: if update process hangs > 60s, ECU rolls back to previous SW
- Cryptographic verification: ECDSA-256 signature on each firmware image before activation

## 6. Cybersecurity Implementation

### Threat Model (TARA Summary)
| Asset | Threat | Control |
|-------|--------|---------|
| Torque commands (CAN FD) | Message spoofing → unintended acceleration | **SecOC** AES-128-CMAC authentication |
| BMS charge commands | Replay attack → battery overcharge | SecOC + **Freshness Value counter** |
| OTA firmware | Malicious firmware → ECU corruption | **ECDSA-256 code signing** |
| TCU remote access | Remote code execution via 4G | Gateway firewall + **TLS 1.3** |
| UDS diagnostic access | Unauthorized calibration write | **UDS Security Access** (0x27) seed-key |
| XCP calibration | Parameter manipulation | **XCP seed-key** + access level restriction |

### SecOC on Critical CAN FD Messages

- VCU torque demand to MCU: SecOC with 24-bit truncated CMAC + 8-bit freshness counter
- BMS contactor command: SecOC with full 128-bit CMAC (safety-critical, bandwidth available on CAN FD)
- EPS steering command from ADAS: SecOC, ASIL D path

### HSM (Hardware Security Module)
- Infineon AURIX TC3xx / NXP S32 series ECUs include **integrated HSM core**
- Keys provisioned during ECU manufacturing (key injection station)
- Key never accessible outside HSM; all crypto operations execute inside HSM

## 7. Functional Safety (ISO 26262)

### ASIL Assignment

| Function | ASIL | Rationale |
|----------|------|-----------|
| AEB autonomous braking | **ASIL D** | Unintended braking at speed = potentially fatal |
| EPS torque overlay (LKA) | **ASIL D** | Loss of steering control |
| Battery contactor control | **ASIL D** | HV exposure risk, fire risk |
| BMS cell overvoltage protection | **ASIL C** | Battery fire risk |
| Motor torque command | **ASIL C** | Unintended acceleration |
| Instrument cluster warning lamps | **ASIL B** | Safety information display |
| TPMS/low tire warning | **ASIL A** | Driver information |
| Infotainment display | **QM** | Non-safety |

### Safety Architecture for AEB (ASIL D example)

```
Requirement: "AEB shall not apply autonomous brake unintentionally"

→ Safety mechanism: Dual-channel object detection agreement
   Channel 1: Camera object detection
   Channel 2: Radar object detection
   Activation: ONLY when BOTH channels agree on obstacle type + TTC < threshold

→ Hardware redundancy: Two independent ECU partitions in DCU
→ Diagnostic coverage: Watchdog on sensor data freshness (< 100ms)
→ Verification: Monitor mode exit condition — driver braking > system threshold
→ Degradation: Sensor fault → functional degrade (no AEB but FCW warning)
→ FMEA: Loss of radar → AEB disabled, FCW camera-only, driver notified
```

### SOTIF (ISO 21448) Considerations

- **Known unsafe scenarios:** Poor visibility (fog, heavy rain) → AEB detection range reduced → SOTIF mitigation: reduce speed threshold
- **Unknown scenarios:** Novel pedestrian clothing / unusual lighting angles
- BYD DiPilot validation: > 100M km simulation + > 5M km real-world fleet data before release
