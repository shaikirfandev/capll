## Scenario 11 — Navigation Re-Routes to Toll Roads When "Avoid Tolls" is Enabled

> **Interviewer:** "A customer has 'Avoid Toll Roads' setting enabled in navigation, but the system keeps routing them through toll roads. How do you investigate this?"

**Investigation Path:**

**Step 1 — Reproduce with specific route:**
- Identify the exact start/end pair where toll roads appear in route
- Record: GPS coordinates of start, end, and toll point
- Compare: route generated by IVI vs route from reference navigation system (Google Maps) with same setting

**Step 2 — Map data check:**
- The road segment marked as toll must have the correct attribute in the map database
- Check if the specific toll road segment is correctly tagged as `toll=yes` in the map data (HERE/TomTom)
- New toll roads: if toll was activated after the map data was last updated → route engine doesn't know it's a toll road

**Step 3 — Setting persistence:**
```bash
adb shell settings get secure navigation_avoid_tolls
# Expected: 1 (avoid)
# If 0: setting not persisting after restart → NVM write bug
```

**Step 4 — Route algorithm priority:**
Some navigation systems have a hierarchy:
- Fastest route takes priority over toll avoidance if toll adds > 20 min
- This can appear as a bug to users
- Check: is there a "fastest even with tolls" override threshold in config?

**Step 5 — Multiple routing engines:**
Some IVI systems use online routing (server-based) for connected mode and offline routing (on-device map) for no-network mode.
Test: disable data connection → reroute → does toll avoidance work?
If works offline but not online: the online routing server is ignoring the toll preference flag sent from the IVI.

**CAPL — Verify navigation preference CAN signals:**
```capl
on message Navigation::RoutePreferences_BC {
  write("Nav Prefs: AvoidTolls=%d  AvoidHighways=%d  AvoidFerries=%d  RouteType=%d",
        this.Nav_AvoidTolls,
        this.Nav_AvoidHighways,
        this.Nav_AvoidFerries,
        this.Nav_RouteType);    // 0=fastest 1=shortest 2=eco
}
```

**Test Cases:**
```
TC_NAV_TOLL_001: Avoid tolls ON → route between known city pair → confirm no toll road in route
TC_NAV_TOLL_002: Avoid tolls OFF → same city pair → confirm toll road is offered as option
TC_NAV_TOLL_003: Avoid tolls ON → only route available uses toll → system warns user with option to override
TC_NAV_TOLL_004: Setting persists after: ignition cycle, SW update, factory reset
TC_NAV_TOLL_005: Newly opened toll road (updated map) → correctly avoided when setting ON
TC_NAV_TOLL_006: Avoid tolls + avoid highways both ON → system finds alternate if available
```

**Root Cause Summary:**
Either (1) map database toll attribute missing on specific road segment → raise with map data provider, or (2) connected routing server API call missing `avoid_tolls=true` parameter → SW defect in API composition layer.

## Scenario 12 — Navigation Shows Car Position in the Middle of a Lake

> **Interviewer:** "The car's position dot on the navigation map shows the vehicle in the middle of a lake — even though the car is clearly on a road. What is happening?"

**Investigation Path:**

**Step 1 — GPS raw data quality:**
Check GPS fix quality indicators:
```
HDOP (Horizontal Dilution of Precision): < 2.0 = good, > 5.0 = poor
Number of satellites in view: need ≥ 4 for 3D fix
GPS signal quality per satellite: > 30 dBHz = good lock
```
If HDOP > 5 near a lake (possible reflection multipath) → GPS position unreliable.

**Step 2 — Map Matching algorithm:**
Normally: even if GPS says you're 20m off-road, the navigation system "snaps" the position to the nearest road using a map-matching algorithm.
If position is shown in a lake: map matching is failing → either no road nearby in the database, or map matching disabled.

**Step 3 — Map data completeness:**
- Is this an area with recently added roads not in the current map version?
- If the car is on a new road not in the map → map matching cannot snap to any road → GPS raw position shown → appears to be off-road
- Check: when was the map last updated? Is there a newer road in that area?

**Step 4 — GPS multipath reflection:**
Lakes and large flat surfaces create GPS multipath:
- GPS signal bounces off water surface → receiver gets two copies of signal → position error
- Position error can be 50–200m in severe cases → puts position in the lake

**Step 5 — Dead reckoning override:**
If vehicle speed = 30 km/h and GPS says "position moved 200m sideways in 0.5s" (impossible acceleration), DR should override GPS.
Test: is DR active? If not, the position jump filter is not working.

**Test Cases:**
```
TC_POS_001: Drive along known road → position snaps to road within 10m at all times
TC_POS_002: Drive through area with known GPS multipath (tunnel exit, near lake) → position recovers within 5s
TC_POS_003: GPS signal fully blocked (RF shield) → position uses DR for up to 60s without excursion
TC_POS_004: Map-matching with new road not in database → position shown at start of known road (fallback)
TC_POS_005: Position jump > 50m in < 1s → flagged as GPS outlier, position not updated
TC_POS_006: Map update installed → new road immediately available for map matching
```

**Root Cause Summary:**
GPS multipath from the lake surface + map matching failure because there is no road to snap to in the map database. The combination means raw erroneous GPS coordinates are shown. Fix: improve outlier rejection in GPS pre-processor (ignore positions that require > 100m/s implied speed).

## Scenario 13 — Navigation Voice Prompts in Wrong Language After Country Border Crossing

> **Interviewer:** "A driver crosses from Germany into France. Navigation map automatically switches to French language road names — correct. But voice prompts continue in German. How do you investigate?"

**Investigation Path:**

**Step 1 — Understand the system architecture:**
Two separate components:
- **Map display language:** controlled by the map app, switches based on locale
- **Text-to-Speech (TTS) engine:** separate component, reads out street names and instructions

The bug: map app locale updated, TTS engine locale not.

**Step 2 — Check TTS locale setting:**
```bash
adb shell settings get secure tts_default_locale
# Expected after crossing: fr_FR
# If still: de_DE → TTS locale not updated when map locale changed
```

**Step 3 — Auto-locale trigger:**
Navigation map uses GPS position to determine country → changes display language.
But TTS locale change may require a separate trigger or may be controlled by the IVI system locale (not the map app locale).

**Step 4 — Configuration check:**
```bash
adb shell getprop persist.sys.locale    # system-wide locale
adb shell getprop persist.sys.language  # language component
```
If system locale is still `de_DE` after border crossing: the navigation app changed only its own display locale but did not update the system TTS locale — an API integration gap.

**Step 5 — Manual test:**
```bash
# Force TTS locale to French and observe navigation
adb shell settings put secure tts_default_locale fr_FR
# Then trigger a navigation voice prompt
adb shell am broadcast -a android.navigation.TEST_VOICE_PROMPT
```

**Test Cases:**
```
TC_TTS_001: Start navigation in Germany (German TTS) → drive into France → TTS switches to French within 500m of border
TC_TTS_002: User manually overrides TTS language to English → does not change when crossing border
TC_TTS_003: Multiple border crossings in 1 hour (DE→FR→CH→DE) → TTS follows each transition correctly
TC_TTS_004: TTS language change does not affect music playback language or UI language (independent)
TC_TTS_005: TTS French voice reads French street names correctly (e.g., Champs-Élysées, Île-de-France)
TC_TTS_006: No voice prompt lag during language switch transition (< 2s delay)
```

**Root Cause Summary:**
Map application uses an internal locale override for display but calls TTS via Android's system TTS API which uses the system locale. When map's display locale changes, it does not call `Locale.setDefault()` or update the TTS engine — the TTS service keeps using the last initialised locale. Fix: on country change event, re-initialise the TTS engine with the new locale.

## Scenario 14 — Speed Camera Alerts Triggering 500m Before Actual Camera

> **Interviewer:** "Navigation is alerting the driver to a speed camera 500m before the camera is actually present. This is causing false alarms and driver distraction. What do you investigate?"

**Investigation Path:**

**Step 1 — Confirm the alert distance spec:**
- What is the specified alert distance for this vehicle market?
- EU: typically 150–300m advance warning
- US: typically 300–500m (or disabled per local law)
- If the alert at 500m is within spec for the market → not a bug, a requirement clarification needed

**Step 2 — GPS position accuracy:**
- If the car's GPS reports the car is 500m from the camera but actually 1000m away → GPS position is 500m ahead of actual position
- GPS position error of 500m is very large → check HDOP, satellite count

**Step 3 — Speed camera database accuracy:**
- The camera location in the POI database may be mis-tagged by 500m
- Check: what are the exact GPS coordinates of the camera in the database vs physical location?
- Tool: measure actual camera coordinates with a reference GPS device (phone)
- Compare vs map database coordinates

**Step 4 — Alert trigger distance formula:**
```
Alert_Distance = Base_Alert_Distance + (Speed × Reaction_Factor)
At 130 km/h: Alert_Distance = 150m + (130 × 2.0) = 410m
```
If `Reaction_Factor` was recently changed from 1.0 to 2.0 in a SW update → alerts now trigger earlier.

**Step 5 — Moving average position:**
Some navigation systems smooth the GPS position with a moving average to reduce jitter.
This effectively places the displayed position 2–3 seconds behind actual position.
At 130 km/h, 2.5s lag = 90m behind → but 500m discrepancy is too large for this.

**Test Cases:**
```
TC_CAM_001: Approach known speed camera at 90 km/h → alert triggers at 300m ± 50m
TC_CAM_002: Approach same camera at 50 km/h → alert triggers at same 300m (not speed-dependent)
TC_CAM_003: Pass camera → alert clears within 50m after passing camera position
TC_CAM_004: Two cameras 400m apart → two separate alerts with correct positions
TC_CAM_005: Camera database update → new cameras alert correctly, removed cameras no longer alert
TC_CAM_006: Alert distance correct in multiple markets: DE (300m), UK (300m), US (disabled)
```

**Root Cause Summary:**
Most likely the speed camera coordinates in the POI database are incorrect — the database entry was geo-coded to the road intersection rather than the camera physical position. Raise with the map/POI data provider with the specific camera GPS coordinates.

## Scenario 15 — Navigation Reroutes Incorrectly After Going Straight at a Junction

> **Interviewer:** "At a complex junction where the road splits into three, the driver follows navigation instructions and goes straight. Navigation immediately says 'performing U-turn' and tries to route back. The driver was on the correct road. What happened?"

**Investigation Path:**

**Step 1 — Understand the junction geometry:**
- Complex junctions (motorway interchanges, spaghetti junctions) have roads very close together
- GPS accuracy (±5–10m) may be insufficient to determine which of three roads the car is on

**Step 2 — Map matching confidence:**
After a junction, the map matcher must decide which road the car is on:
- It evaluates: heading alignment, speed, road attributes (speed limit, lane count)
- If confidence < threshold → "best guess" road selection
- If wrong road selected → navigation thinks car is on road A, car is actually on road B → mismatch → U-turn recommended

**Step 3 — GNSS heading accuracy:**
At slow speed (< 20 km/h) through a complex junction, GPS heading is unreliable.
If car goes straight but GPS heading shows 15° turn → map matcher selects a slightly angled road by mistake.

**Step 4 — Lane-level navigation:**
Modern navigation has lane-level precision using HD maps.
If HD map is not available for this junction → navigation uses road-level precision → more likely to misidentify the exit taken.

**Step 5 — Reproduce and log:**
```python
# Use FastF1 pattern applied to GPS log
import pandas as pd
import numpy as np

# Load navigation GPS log (exported from IVI)
gps_log = pd.read_csv('junction_gps_log.csv')   # timestamp, lat, lon, heading, speed, matched_road_id

# Find the junction point
junction_idx = gps_log[gps_log['lat'].between(25.2050, 25.2055)].index
print(gps_log.iloc[junction_idx - 2: junction_idx + 5][['timestamp','lat','lon','heading','matched_road_id']])
# If matched_road_id changes unexpectedly after junction → map matching error
```

**Test Cases:**
```
TC_JUNC_001: Drive through 3-way junction taking middle exit → navigation continues without reroute
TC_JUNC_002: Drive through junction taking wrong exit → navigation reroutes correctly within 200m
TC_JUNC_003: Complex motorway interchange at 100 km/h → position maintained correctly
TC_JUNC_004: Junction covered by HD map → position accuracy ≤ 1m lane width
TC_JUNC_005: Junction with no HD map coverage → position accuracy ≤ 10m (road level)
TC_JUNC_006: 50 consecutive junction passages → no false reroute events logged
```

**Root Cause Summary:**
Map matching algorithm uses only GPS position without sufficient weighting of speed vector and road heading at low junction speeds. At the specific junction, three roads have very similar headings (within 15° of each other), so heading alone cannot disambiguate. Fix: incorporate gyroscope yaw rate data into map matching confidence scoring.
